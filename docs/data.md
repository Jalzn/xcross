# Data

Data folder for `xcross`. The whole `data/` directory is git-ignored: PFF tracking data is
licensed and must not be redistributed. Obtain it yourself and place it under `data/raw/`.

It is split into two trees:

```
data/
  raw/        # PFF originals (read-only for the pipeline)
  processed/  # parquet tables generated by the processing module
```

The internal structure mirrors on both sides:

```
<root>/<league>/<season>/<match_id>/<files>
```

E.g. `data/raw/premier-league/2024-2025/31995/` and
`data/processed/premier-league/2024-2025/31995/`.

## `raw/`

Raw PFF files, one set per match. Per match:

- `<match_id>.jsonl.bz2` — one JSON line per tracking frame (~25 fps). Each frame carries
  smoothed positions of the players (home/away) and the ball, plus the `game_event` and
  `possession_event` fields when the frame corresponds to an event.
- `metadata.json` — match info: teams, date, pitch dimensions, fps, `homeTeamStartLeft`
  (which side the home team starts attacking).
- `rosters.json` — line-ups with `player_id`, `team_id`, shirt number, position (and the
  shirt ↔ player_id relation needed to associate tracking with players).

The full vendor format specification ships with the licensed PFF FC data and is not
redistributed here; the fields this pipeline relies on are the ones listed above.

`raw/` is **immutable** for the pipeline. Do not write anything here.

## `processed/`

Generated by `uv run python -m xcross.data.build`. Per match:

- `crosses.parquet` — one row per cross (PFF `CR` marker) that passed **two geometric
  filters**: it started in the **origin zone** (final attacking third and out wide on the
  flank, outside the penalty-box width) and the ball reached the **target zone** (opposing
  penalty box). This discards central passes, balls played from the back, and corners (which
  PFF also tags as `CR`, but start in the corner). The zones are derived from each match's
  real pitch dimensions (`meta.parquet`).
- `tracking.parquet` — player positions frame by frame, **only** within each cross window.
  The window runs from the `CR` frame to the first of: next possession event, ball out of
  play (`OUT`), or a 5 s cap. Coordinates are already **standardised**: the crossing team
  always attacks toward `+x`.
- `ball.parquet` — ball x, y, z positions per frame, same windows and same standardisation.
- `meta.parquet` — one row with the match metadata (to avoid re-parsing `metadata.json` in
  analyses).
- `roster.parquet` — one row per rostered player.

`processed/` can be deleted and rebuilt at any time — all information derives from `raw/`.

## Regenerate

```bash
uv run python -m xcross.data.build                       # all matches
uv run python -m xcross.data.build --leagues premier-league
uv run python -m xcross.data.build --matches 31995 --rebuild
```
